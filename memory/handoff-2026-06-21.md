## TL;DR

Theme of the recent work: **a venue order rejection must never be
silent.** On a real-money account a bare `(rejected)` / `Inactive` with
no cause is unacceptable. Four commits on `claude/happy-cori-eno5ux`
(ahead of `master`) lift the venue's reject reason all the way from the
IBKR TWS callbacks → `OperationResult.error` → every Alice-facing
surface (order history, push summary, status log line). **Observability
only — no order-placement control flow changed.**

## What changed, precisely (so you don't re-derive it)

### 1. The chokepoint — `TradingGit.ts`

- **`parseOperationResult`** is the broker-agnostic chokepoint. A venue
  reject arrives as `success: true` + `OrderState.status === 'Inactive'`
  — **never as a thrown error**. The cause lives on
  `orderState.rejectReason` / `orderState.warningText` (IBKR; Alpaca uses
  `rejectReason` too). New `orderStateReason()` helper joins those fields
  (trimmed, empty-string defaults dropped) and, when `status === 'rejected'`,
  lifts the reason into the first-class **`OperationResult.error`**.
- **`formatOperationChange`** (placeOrder + closePosition status lines)
  now appends `: <reason>` onto a `rejected` status instead of a bare
  `(rejected)` / `close (rejected)`.
- Net effect downstream: order-history projection and `inbox_push`
  summaries that already read `result.error` / `result.status` now show
  the *why* for free.

### 2. The source — IBKR `request-bridge.ts`

- Added `REQ_ID_BASE = 10_000` constant (was a magic number). reqIds are
  allocated from 10_000 up; TWS **order ids live below that** (handed out
  by `nextValidId`). The gap is what lets `error()` distinguish an order
  reject from a data-request failure when only an id is in hand.
- **`openOrder`**: new `logOrderOutcome()` emits one status line carrying
  `status + rejectReason/warningText + contract/order context` (action,
  qty, symbol, secType/currency). Rejects (`status 'Inactive'`) **always
  log at `warn`**; non-reject statuses log only on the
  placeOrder/modifyOrder path (`verbose`) so the `reqOpenOrders` poll
  batch stays quiet. `orderState` still flows on to the placeOrder result
  unchanged.
- **`error()`**: an order-range id (`< REQ_ID_BASE`) with no pending
  request means the order already resolved via an `Inactive openOrder`
  ahead of this trailing `error()` — `warn` with the venue reason instead
  of swallowing it.

Result a user now sees:
`Inactive — rejectReason: "Order rejected - reason: forex trading is not
allowed for this account"` instead of a bare `Inactive`.
